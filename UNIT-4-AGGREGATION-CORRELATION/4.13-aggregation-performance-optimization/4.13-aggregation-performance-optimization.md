## 📚 CORE CONCEPTS

### Performance Metrics

**Metrics to track:**

1. **Total Time:** How long aggregation takes end-to-end
 - Target: < for 13 users
 - Large sources: Can take 

2. **Objects Read:** Users, groups, accounts processed
 - Entra ID: 13 users, 7 groups = 20 objects minimum

3. **Objects Stored:** How many got saved to ISC
 - Target: 100% (all read objects stored)

4. **Errors:** How many failed
 - Target: 0

5. **Throughput:** Objects per second
 - Entra ID small: 5-10 objects/sec typical
 - Large sources: 100-500 objects/sec

---

### Performance Bottlenecks

**Bottleneck 1: Slow Network Connection**

**Symptom:** Aggregation takes 30+ minutes for 13 users (should be <5 min).

**Cause:** Network latency between ISC and Entra ID.

**Fix:**
- Check ISC location (region)
- Check Entra ID location
- If far apart, consider regional ISC deployment
- Verify no network throttling/firewalls

---

**Bottleneck 2: Source System Under Load**

**Symptom:** Aggregation runs slow, source system shows high CPU.

**Cause:** Source system (Entra ID) struggling with API requests.

**Fix:**
- Schedule aggregation during off-peak hours (nights, weekends)
- Use partial aggregation (not full) between peak times
- Limit objects read per aggregation (if connector supports)
- Contact source vendor if persistently slow

---

**Bottleneck 3: ISC Server Under-Resourced**

**Symptom:** Aggregation starts but ISC becomes unresponsive.

**Cause:** ISC server out of memory or CPU maxed.

**Fix:**
- Check ISC server resources (Administration > System Health)
- Increase server CPU/memory (contact ISC admin)
- Schedule aggregation when ISC not in heavy use
- Break large aggregations into smaller chunks

---

**Bottleneck 4: Database Performance**

**Symptom:** Aggregation stuck at "storing data" phase.

**Cause:** ISC database slow, many disk I/O operations.

**Fix:**
- Check database disk space (should have >20% free)
- Run database maintenance tasks (if ISC admin available)
- Increase database timeout (Administration > Database settings)
- Upgrade database server if persistently slow

---

**Bottleneck 5: Attribute Transformation Overhead**

**Symptom:** Aggregation slow specifically when transformations enabled.

**Cause:** Complex attribute mapping rules being evaluated for each object.

**Example:** Transformation rule: "concatenate firstName + lastName + hire year"
- Simple mapping: Quick
- Complex transformation: Slower (evaluated per user)

**Fix:**
- Simplify transformations if possible
- Use direct attribute mapping instead of calculated fields
- Cache transformation results (if available)
- Profile transformations to find expensive ones

---

### Optimization Strategies

**Strategy 1: Use Partial Aggregation**

```
Schedule:
 Monday 2am: Full aggregation (all users)
 Tue-Sun 2am: Partial aggregation (changes only)
 Result: Most days fast, accuracy maintained by weekly full
```

**Benefits:** 2x-10x faster than full, catches daily changes
**Drawback:** Doesn't detect deletions

---

**Strategy 2: Filter Aggregation Scope**

**If source has 10,000+ users but only 100 needed:**

```
Account mapping filter: "Department = Engineering"
Result: Only 100 users read and stored, not 10,000
```

**Benefits:** Much faster, less disk space, cleaner data
**Drawback:** Other departments not available

---

**Strategy 3: Parallelize Aggregations**

**If multiple sources (Entra ID, Okta, Oracle):**

```
Schedule all 3 to run at same time (different connectors)
Result: 3x faster than running sequentially
ISC server must have resources (CPU, memory)
```

**Benefits:** Overall completion faster
**Drawback:** Requires careful scheduling, uses more resources

---

**Strategy 4: Off-Peak Scheduling**

```
Schedule aggregation during:
 - Late night (2-4am) when users not working
 - Weekends
 - Planned maintenance windows

Avoid:
 - Business hours (9am-5pm)
 - End of month (heavy reporting)
 - Known peak times for source system
```

**Benefits:** No user impact, source system has capacity
**Drawback:** May delay discovery of issues (if problems happen during day)

### For Contoso Setup (Small Dataset)

**13 users and 7 groups = small dataset:**

- Full aggregation: ~30-60 seconds
- Partial aggregation: ~10-15 seconds
- Performance not a concern at this scale

**Optimization**: Not needed for test, but understand concepts for production.

### Performance Monitoring

**ISC aggregation logs show:**
```
Aggregation started: 2024-03-02 02:00:00
Objects discovered: 20 (13 users, 7 groups)
Objects read: 20
Objects stored: 20
Errors: 0
Time taken: 45 seconds
Throughput: 0.44 objects/sec
Status: SUCCESSFUL
```

**Monitor these metrics over time:**
- Is time increasing? (May indicate growing dataset or degrading performance)
- Error rate? (Should always be 0)
- Throughput? (Should be consistent)

- Schedule off-peak to avoid user impact
- Monitor metrics over time for trends

## 🎓 CERTIFICATION

**Q:** If aggregation takes to complete for 13 users (expected ~1 min), what is the first thing to check?

A) ISC database corruption
B) ✅ Network connectivity and latency between ISC and source
C) Source has more than 13 users
D) Aggregation rule syntax error

**Answer: B.** If much slower than expected, likely network latency issue. Test connection and check network path.

**Q:** When would you use partial aggregation instead of full?

A) Always (it's faster)
B) Only when errors occur
C) ✅ For daily/frequent updates to catch changes quickly
D) Never (full is more reliable)

**Answer: C.** Partial faster for frequent runs, full better for consistency/weekly. Hybrid approach (daily partial, weekly full) is best.
