## Key Concepts

### Multi-Tenant ISC Architecture

**Multi-Tenancy Definition:** Single ISC instance serving multiple independent organizations (customers)

**Multi-Tenant Requirements:**
- **Data Isolation:** Customer A's data never visible to Customer B
- **Metadata Separation:** Customer A's policies, roles, workflows independent
- **Audit Isolation:** Each tenant's audit logs separate
- **Performance:** One customer's load doesn't impact others
- **Customization:** Each tenant can customize ISC without affecting others

**Multi-Tenant Challenges:**
- Complex database design (logical vs. physical separation)
- Resource contention and noisy neighbor problems
- Compliance complexity (GDPR for tenant in EU, HIPAA for tenant in healthcare)
- Upgrade coordination (all tenants affected)
- Backup/restore complexity (granular recovery per tenant)

### High Availability (HA) Concepts

**HA Goal:** Zero (or near-zero) downtime through redundancy

**HA Components:**
```
Redundancy Level 1: Multiple Application Servers
  - Primary ISC instance: US-East
  - Secondary ISC instance: US-West (standby)
  - Load balancer directs traffic to primary

Redundancy Level 2: Database Replication
  - Primary database: US-East
  - Read replica: US-West (real-time sync)
  - Automatic failover if primary fails

Redundancy Level 3: Data Center Failover
  - Primary data center: US-East (active)
  - Secondary data center: US-West (hot standby)
  - Automatic traffic rerouting
```

**HA Metrics:**
- **Availability:** % of time system is operational
  - 99.9% (three nines) = 43 minutes downtime/month
  - 99.99% (four nines) = 4 minutes downtime/month
  - 99.999% (five nines) = 26 seconds downtime/month

**RTO (Recovery Time Objective):** How long before system back online
- HA RTO: < 1 minute (automatic failover)
- DR RTO: 4-8 hours (manual intervention)

**RPO (Recovery Point Objective):** How much data loss acceptable
- HA RPO: < 1 minute (near real-time replication)
- DR RPO: 1-4 hours (periodic backups)

### Disaster Recovery (DR) Concepts

**DR vs. HA:**
| Aspect | HA | DR |
|--------|----|----|
| **Scope** | Single data center | Multiple data centers |
| **Cause** | Hardware failure | Region-wide outage, sabotage |
| **Recovery** | Automatic | Manual + automated |
| **Time** | Seconds to minutes | 4-8 hours |
| **Cost** | High (redundant infrastructure) | Medium (warm/cold standby) |
| **Testing** | Monthly | Quarterly minimum |

**DR Strategies:**

1. **Backup & Restore (Cold DR)**
   - Most cost-effective
   - RPO: 4-24 hours
   - RTO: 4-8 hours
   - Use: Non-critical systems

2. **Warm Standby DR**
   - Secondary site updated periodically
   - RPO: 1 hour
   - RTO: 2-4 hours
   - Use: Important systems

3. **Hot Standby DR (Active-Active)**
   - Both sites active simultaneously
   - RPO: < 1 minute
   - RTO: < 5 minutes
   - Use: Critical systems
   - Cost: 2x infrastructure

#### Step 3: Configure Standby ISC Instance and Replication

**Action:** ISC Administration → Infrastructure → New Standby Instance

**Configuration:**
```
Standby Instance Setup:
  Name: ISC-Standby-US-East
  Region: us-east-1
  Availability Zone: us-east-1b (different from primary)
  Instance Type: Large (matching primary)
  Database: PostgreSQL 14 (Replica)
  Status: STANDBY (not receiving traffic)

Database Replication Setup:
  Mode: Synchronous replication
    - All writes confirmed on standby before commit
    - Primary waits for standby acknowledgment
    - Highest data safety, slight performance cost

  Replication User: replication@standby
  Replication Slot: wal_receiver_standby

  Standby Configuration:
    hot_standby = on (can accept connections)
    hot_standby_feedback = on (prevent vacuum conflicts)

Expected Result:
  ✓ Standby instance launched
  ✓ Database replication initialized
  ✓ Replication status: In Sync
  ✓ Standby ready for automatic failover
```

**Verification:**
```
Check Replication Status (on primary):
  SELECT * FROM pg_stat_replication;

Expected output:
  pid | usesysid | usename      | application_name | write_lsn | flush_lsn | replay_lsn
  --- | -------- | ------------ | ---------------  | --------- | --------- | ----------
  123 | 16384    | replication  | standby          | 0/30000F0 | 0/30000F0 | 0/30000F0
  (Row shows replication in sync)

Check Standby Status (on standby):
  SELECT pg_is_in_recovery();
  Result: t (true - standby is in recovery mode)
```

#### Step 10: Document Recovery Runbook

**Action:** Create documented recovery procedures

**Runbook Template:**
```
ISC HA/DR Recovery Runbook

1. Automatic Failover (HA Event)
   Trigger: Primary ISC unavailable
   Timeline: < 5 minutes (automatic)
   Responsibility: Automated (no human action needed)

   Detection:
     - Load balancer health check fails twice (10 seconds)
     - Traffic automatically reroutes to standby

   Verification:
     - Check load balancer: Primary = UNHEALTHY, Standby = HEALTHY
     - Verify service: Can access isc.contoso.com
     - Check application: ISC console responsive

   Escalation: If still down after 15 minutes, page on-call DBA

2. Manual Failover (DR Event)
   Trigger: Primary data center completely down
   Timeline: 2-4 hours
   Responsibility: DR Team

   Steps:
     a) Declare incident (call incident bridge)
     b) Activate secondary site (US-West)
     c) Restore database from backup
     d) Test application functionality
     e) Update DNS to secondary
     f) Notify customers
     g) Begin primary recovery

3. Complete Disaster Recovery
   Trigger: Both primary and secondary down
   Timeline: 6-8 hours
   Responsibility: DR Team + Management approval

   Steps:
     a) Provision new ISC instance (tertiary location)
     b) Restore from daily backup (12 hours old max)
     c) Apply transaction logs (recover changes)
     d) Verify all critical systems
     e) Activate recovered instance
     f) Comprehensive customer notification
     g) Post-incident review scheduled
```

**Expected Result:**
```
✓ Runbook documented with step-by-step procedures
✓ Escalation paths defined
✓ Success criteria documented
✓ Contact list updated
✓ Runbook stored in wiki and printed copies available
```

## Key Takeaways

✓ **HA and DR are different** - HA prevents downtime, DR recovers from disasters
✓ **Plan for RTO and RPO** - Define acceptable downtime and data loss upfront
✓ **Test regularly** - Untested procedures fail when you need them most
✓ **Automate failover** - Human manual steps are slow and error-prone
✓ **Document everything** - Runbooks are essential during incidents
✓ **Multi-tenant requires isolation** - Data leakage is unacceptable
✓ **Replication is critical** - Standby must have current data to be useful
