### Project Overview

```
Company Name:       MedTech Solutions Inc.

Current State:      Legacy identity system (no automation)

```

### Business Context

**Company Background:**

MedTech Solutions is a leading cloud-based medical records and care coordination platform. Hospitals and clinics use their platform to manage patient records, schedule care, and coordinate between departments and specialists.

```
Success Drivers:
в”ңв”Җ Patient safety (secure access control)

```

### Current Challenges

**Challenge 1: Manual Access Control (3-5 day turnaround)**

```
Current Workflow:
в”ңв”Җ Clinician hired at hospital
в”ңв”Җ Hospital admin emails MedTech: "Please add Dr. John Smith"
в”ңв”Җ MedTech support staff: Manually creates account
в”ңв”Җ MedTech support staff: Assigns access to hospital
в”ңв”Җ MedTech support staff: Sends credentials to hospital
в”ңв”Җ Time elapsed: 3-5 business days
в”ңв”Җ Cost: $50-75 per user creation
в”ңв”Җ Errors: 5-10% of accounts have wrong access
в””в”Җ Problem: Patient care delayed, clinicians frustrated

With ISC (Target):
в”ңв”Җ Hospital admin: Self-service provisioning
в”ңв”Җ ISC: Automated account creation
в”ңв”Җ ISC: Automated access assignment
в”ңв”Җ ISC: Automated credential delivery
в”ңв”Җ Time elapsed: < 2 hours
в”ңв”Җ Cost: < $2 per user creation (99% automation)
в”ңв”Җ Errors: < 0.1% with automated validation
в””в”Җ Result: Clinician can start seeing patients immediately
```

**Challenge 2: No Automated Deprovisioning (Security Risk)**

```
Current Problem:
в”ңв”Җ Clinician leaves hospital
в”ңв”Җ Hospital doesn't notify MedTech
в”ңв”Җ Former clinician still has access to patients
в”ңв”Җ HIPAA violation: Patient data exposed to unauthorized user
в”ңв”Җ MedTech only discovers during annual audit
в”ңв”Җ Risk: Patient safety issue, compliance violation, $50K+ fine per violation
в””в”Җ Estimated risk: 1% of former users have access (5,000 exposed users)

With ISC (Target):
в”ңв”Җ Clinician terminates employment
в”ңв”Җ Hospital updates ISC (automated from HRIS)
в”ңв”Җ ISC: Deprovisioning triggered automatically
в”ңв”Җ ISC: Patient data access revoked immediately
в”ңв”Җ Compliance: Audit-ready deprovisioning trail
в””в”Җ Result: Patient safety maintained, compliance verified
```

**Challenge 3: Compliance Audit Complexity (12 weeks per audit)**

```
Current Audit Process:
в”ңв”Җ Regulator: "Show us who has access to patient data"
в”ңв”Җ MedTech: Manual report of 500,000 users
в”ңв”Җ MedTech: Manual verification (3 weeks)
в”ңв”Җ Regulator: Questions about specific users
в”ңв”Җ MedTech: Manual investigation (2 weeks)
в”ңв”Җ Regulator: Findings and remediation requests

в”ңв”Җ Cost: $200K+ (staff time, auditor fees)
в””в”Җ Result: Stressful, error-prone, expensive

With ISC (Target):
в”ңв”Җ Regulator: "Show us who has access to patient data"
в”ңв”Җ ISC: One-click compliance report generated
в”ңв”Җ Report: Complete, audit-ready, 100% accurate
в”ңв”Җ Regulator: Spot checks user access
в”ңв”Җ ISC: Real-time verification of access rules
в”ңв”Җ Compliance: All rules enforced automatically
в”ңв”Җ Total audit duration: 1-2 weeks
в”ңв”Җ Cost: $20K (minimal staff, regulator satisfied)
в””в”Җ Result: Quick, confident, compliant
```

### ISC Solution Design

**Architecture Decision: Hybrid (Internal + Multi-Tenant)**

```
Architecture:
в”ңв”Җ Internal ISC (1,200 MedTech employees)
в”Ӯ  в”ңв”Җ Single instance, on-premise trust model
в”Ӯ  в”ңв”Җ Integration with MedTech HR systems
в”Ӯ  в”ңв”Җ HA/DR for reliability (24x7 operations)
в”Ӯ  в””в”Җ Audit trail for SOX compliance
в”Ӯ
в””в”Җ Multi-Tenant ISC (150 customers, 500,000+ users)
   в”ңв”Җ Separate instance from internal ISC
   в”ңв”Җ Dedicated database and infrastructure
   в”ңв”Җ Each customer: Separate tenant or account
   в”ңв”Җ HIPAA-compliant encryption and isolation
   в””в”Җ Automated provisioning and deprovisioning

Technology Stack:
в”ңв”Җ Identity Platform: SailPoint ISC (cloud-native)
в”ңв”Җ Database: PostgreSQL (replicated for HA)
в”ңв”Җ Connectors: 30+ healthcare system integrations
в”Ӯ  в”ңв”Җ HR: Workday, ADP
в”Ӯ  в”ңв”Җ EHR: Epic, Cerner, Athena
в”Ӯ  в”ңв”Җ Hospital systems: Various proprietary
в”Ӯ  в””в”Җ Cloud: Azure AD, Okta
в”ңв”Җ Automation: Workflows, roles, policies
в”ңв”Җ Analytics: Compliance dashboards, user metrics
в””в”Җ Security: Encryption, audit logs, HIPAA controls
```

**Compliance & Security Framework:**

```
HIPAA Requirements:
в”ңв”Җ Encryption at rest: AES-256 (ISC native)
в”ңв”Җ Encryption in transit: TLS 1.3 (all APIs)
в”ңв”Җ Access control: RBAC (role-based)
в”ңв”Җ Audit trail: Complete, immutable (7-year retention)
в”ңв”Җ Data segregation: Per-customer isolation
в”ңв”Җ Backup/Recovery: Daily backups, <4 hour RTO
в”ңв”Җ Penetration testing: Quarterly by third party
в””в”Җ Compliance status: Annual audit HIPAA-ready

HITECH Act (Breach Notification):
в”ңв”Җ Incident detection: Automated alerting
в”ңв”Җ Incident response: Documented procedures
в”ңв”Җ User notification: Automated within 60 days
в”ңв”Җ Regulator notification: Automated if >500 users
в””в”Җ Remediation: Documented and executed

SOX Requirements (Public Company):
в”ңв”Җ Access controls: Segregation of duties enforced
в”ңв”Җ Audit trail: CEO/CFO certification ready
в”ңв”Җ Change management: Documented approval process
в”ңв”Җ User access review: Quarterly certification
в””в”Җ Compliance reporting: Monthly automated dashboards

GDPR Requirements (EU Customers):
в”ңв”Җ Data residency: EU data hosted in EU
в”ңв”Җ Data processing: Agreement with EU customer
в”ңв”Җ Right to deletion: Automated deprovisioning
в”ңв”Җ Breach notification: Automated within 72 hours
в””в”Җ DPA (Data Processing Agreement): Standard template
```

**Domain 3: Access Management - Sample Scenarios**

```
Scenario 1: Banking Compliance
A bank manager received an access request from a new employee (Jane) who needs:
- Access to Customer Account System (finance role)
- Access to Compliance Database (read-only)
- Access to Executive Reports (manager role)

Jane's role in the organization: Junior Analyst (entry-level)

Question: Which access should be REJECTED based on SoD (Segregation of Duties)?
A) Customer Account System access
B) Compliance Database read-only
C) Executive Reports (manager role) вҶҗ CORRECT
D) All of the above should be approved

Explanation: A junior analyst should not have manager-level access to
Executive Reports. This violates principle of least privilege.

---

Scenario 2: Contractor Access
A hospital is bringing on a contractor (software developer) to work on their
medical records system. The contractor requests:
- Access to Hospital Network (VPN)
- Access to Database Server (developer role)
- Access to Patient Data (for testing)
- 6-month contract duration

Question: What should be configured in ISC?
A) All access approved for 6 months
B) All access approved for 3 months
C) Network and Database access for 6 months; Patient data for testing only вҶҗ CORRECT
D) Only network access; everything else manual

Explanation: Contractors need time-limited access. Patient data access should
be restricted (HIPAA) and limited to necessary testing only. Access should
auto-expire when contract ends.

---

Scenario 3: Department Transfer
An engineer (Bob) is transferring from Engineering to Sales department.
Current access:
- Engineering systems: GitHub, development tools (5+ systems)
- Office tools: Email, Office 365, Slack (shared)
- Finance: Sales CRM, opportunity management, customer data

Question: What should ISC do automatically?
A) Remove all access immediately
B) Remove Engineering systems; keep Office tools; add Sales CRM вҶҗ CORRECT
C) Keep all existing access and add Sales access
D) Require manual approval for each change

Explanation: ISC should automatically remove department-specific access
(Engineering), keep shared tools, and add new role access (Sales). This is
the "Move" phase of identity lifecycle and should be largely automated.
```

**Domain 5: Enterprise Architecture - Sample Questions**

```
Question 1: A Fortune 500 company with 50,000 users in 20 countries needs
to ensure GDPR compliance while maintaining global consistency. What
architecture pattern is BEST?

A) Single-tenant ISC (all users in one instance)
B) Federated multi-instance (regional ISC in each country) вҶҗ CORRECT
C) Multi-tenant ISC (all users in shared instance)
D) Hybrid with 3 regional instances

Explanation: GDPR requires EU data to stay in EU. Federated architecture
with regional instances ensures data residency compliance while maintaining
global governance consistency through federation layer.

---

Question 2: A company experiences an ISC outage affecting 10,000 users.
Recovery from backup would take 4 hours. What architecture could have
prevented this?

A) Enhanced monitoring and alerting
B) Automated HA failover with RTO < 1 minute вҶҗ CORRECT
C) Daily backups
D) Comprehensive disaster recovery plan

Explanation: HA with automatic failover prevents 4-hour outages entirely.
While backups and DR plans are important, HA provides better protection
for most common failures.
```
