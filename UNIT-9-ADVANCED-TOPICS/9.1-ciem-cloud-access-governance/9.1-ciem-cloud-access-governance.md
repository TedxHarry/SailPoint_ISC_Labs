## Key Concepts

### What is CIEM?

**CIEM (Cloud Infrastructure Entitlement Management)** extends ISC governance to cloud platforms by discovering, analyzing, and managing cloud resource permissions.

**Problem CIEM Solves:**
- Cloud accounts have thousands of permissions (not just on-premises access)
- Users often have overprivileged cloud access (permission creep)
- Cloud permissions are hard to audit and manage
- Compliance frameworks require continuous cloud access governance
- Shadow cloud usage creates security risks

**ISC CIEM Solution:**
```
Discover cloud resources → Analyze permissions → Create policies
→ Detect violations → Remediate automatically
```

### Cloud vs. On-Premises Access Models

| Aspect | On-Premises | Cloud |
|--------|-------------|-------|
| **Access Type** | Role-based groups | Fine-grained permissions (policies) |
| **Complexity** | Simple (100s of roles) | Complex (1000s of permissions) |
| **Change Frequency** | Slow (change management) | Fast (developers self-serve) |
| **Audit** | Group membership | Policy attachment per resource |
| **Remediation** | Remove from group | Detach specific permissions |

### CIEM Discovery Process

**Step 1: Authenticate to Cloud**
```
ISC creates connector to AWS/Azure account
Connector assumes role with read-only permissions
Permissions: "List all IAM roles, users, and policies"
```

**Step 2: Discover Resources**
```
Query cloud provider APIs
Find: Users, roles, permissions, resource access
Example AWS: List IAM users → List attached policies → List inline policies
Result: Complete permission inventory
```

**Step 3: Correlate to ISC Identities**
```
Match cloud user "jsmith@mycompany.com" to ISC identity "Jordan Smith"
Link cloud permissions to ISC user record
Build complete access picture
```

### CIEM Risk Detection

**Risk Category 1: Overprivileged Users**
```
Example: Finance analyst with full AWS admin access
Detection: Permission analysis shows 1000+ permissions when only 5 needed
Risk Level: HIGH
Remediation: Remove admin policy, attach Finance-only policy
```

**Risk Category 2: Unused Permissions**
```
Example: User with S3 write access but never used it
Detection: CloudTrail shows no S3 API calls in 90 days
Risk Level: MEDIUM
Remediation: Remove unused permission
```

**Risk Category 3: Shared Cloud Credentials**
```
Example: Database password shared in team Slack
Detection: Multiple logins from different IPs with same access key
Risk Level: CRITICAL
Remediation: Rotate credentials, implement personal access keys
```

**Risk Category 4: Cross-Account Access**
```
Example: User can assume role in security account
Detection: Unusual cross-account permissions
Risk Level: HIGH
Remediation: Restrict cross-account access to specific roles
```

### Contoso Scenario: Securing Cloud Access

**Problem:**
- Finance team uses AWS for data warehouse
- Developers use Azure for application deployment
- Cloud permissions assigned ad-hoc (no governance)
- Several team members left company but still have cloud access

**ISC CIEM Solution:**
1. Discover all cloud permissions
2. Match to Contoso ISC identities
3. Identify overprivileged users (engineers with finance access)
4. Create policies (developers can only access dev account)
5. Auto-remediate violations
6. Continuous monitoring for changes

### Lab Steps

#### Step 1: Configure AWS Connector

**Action:** ISC Administration → Connectors → New Cloud Connector

**Configuration:**
```
Connector Type: AWS
Name: AWS-Finance-Account
Description: Finance team cloud resources

AWS Account Details:
  Account ID: 123456789012
  Region: us-east-1
  IAM Role: sailpoint-isc-connector-role

Connector Settings:
  Discovery: Enabled (discover all IAM users, roles, policies)
  Frequency: Daily (sync cloud changes)
  Permissions: Read-only (safe to audit)

Expected Result:
  ✓ Connector created
  ✓ AWS credentials validated
  ✓ IAM role permissions confirmed
  ✓ Status: CONNECTED
```

**Troubleshooting:**
```
Error: "Unable to assume role"
  → Verify AWS IAM role trust relationship
  → Confirm role includes proper permissions
  → Check role ARN matches connector config

Error: "Access Denied"
  → Verify connector role has ListUsers, ListPolicies permissions
  → Check if MFA required for role assumption
  → Confirm region matches account region
```

## Key Takeaways

✓ **Cloud access is different** - Permissions-based instead of group-based
✓ **Discovery is first** - Must discover all cloud access before governing
✓ **Correlation matters** - Link cloud accounts to ISC identities for unified governance
✓ **Policies prevent violations** - Automate access control instead of manual reviews
✓ **Monitor continuously** - Cloud permissions change fast, need real-time visibility
✓ **Remediate quickly** - Auto-remediation prevents compliance violations
✓ **Audit everything** - Cloud access is heavily audited by regulators
