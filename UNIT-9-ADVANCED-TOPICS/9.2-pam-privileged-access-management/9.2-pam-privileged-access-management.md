## Key Concepts

### What is PAM?

**PAM (Privileged Access Management)** extends ISC governance to high-risk, high-privilege accounts. PAM adds enhanced controls beyond standard provisioning.

**High-Risk Accounts Include:**
- System administrators (SQL Server DBA, Linux root)
- Cloud platform admins (AWS root, Azure Owner)
- Active Directory admins
- Firewall and network device admins
- Database administrators
- Service accounts with broad permissions

### Why PAM Matters

**Risk:** A single compromised admin account can:
- Delete critical data
- Disable security controls
- Access sensitive business information
- Disrupt operations

**Business Impact:**
- Ransomware attacks start with admin compromise
- Insider threats use admin access for data theft
- Regulatory violations when admin access isn't audited

**ISC PAM Solution:** Layer additional controls on privileged accounts

### PAM Controls in ISC

| Control | Purpose | Example |
|---------|---------|---------|
| **MFA** | Multiple authentication factors | Token + password to access DBA console |
| **Approval** | Explicit approval before access | Manager must approve root access |
| **Session Recording** | Audit admin actions | Record all DBA SQL commands |
| **Time Limits** | Restrict duration | Access expires after 4 hours |
| **Emergency Access** | Break-glass procedures | Emergency root access when normal fails |
| **Monitoring** | Real-time alerts | Alert if DBA accesses personnel database |

### Contoso PAM Scenario

**Problem:**
- Morgan Chen (IT Admin) has SQL Server DBA credentials
- Credentials written on sticky note (security risk)
- No approval required to use admin account
- No audit trail of SQL commands executed

**ISC PAM Solution:**
1. Remove stored credentials
2. ISC becomes credential broker
3. MFA required
4. Manager approval needed
5. Session recorded for audit
6. Access expires automatically
7. Full audit trail created

## Lab: Implement PAM Controls

### Lab Objectives

- Configure privileged account groups
- Create PAM policies
- Test privileged access request workflow
- Review session recordings and audit trails
- Implement emergency access procedures

### Lab Steps

#### Step 1: Identify Privileged Accounts

**Action:** ISC Search → Advanced Filter

**Query:**
```
Type: Service Account OR (Title contains "Admin" OR "DBA" OR "Root")
Status: Active
```

**Expected Result:** List of high-risk privileged accounts

**Sample Results:**
```
1. linux_admin (Linux system admin)
2. db_admin_prod (Production database admin)
3. app_svc_account (Application service account)
4. backup_svc (Backup service account)
5. domain_admin (Active Directory admin)
6. aws_admin (AWS account admin)
7. sql_server_dba (SQL Server DBA)
8. firewall_admin (Network firewall admin)

Count: 10-15 privileged accounts expected
```

#### Step 2: Create Privileged Account Group

**Action:** ISC Administration → Privileged Access → Create Group

**Configuration:**
```
Group Name: Database Administrators
Description: High-risk database admin accounts
Criteria:
  - Account name contains "dba" (case insensitive)
  - Account name contains "database"
  - OR manually selected accounts

Members Found:
- db_admin_prod
- db_admin_dev
- oracle_dba
- mysql_admin
```

**Expected Result:** Group created with 4+ members

#### Step 3: Create MFA Policy

**Action:** ISC Governance → Policies → New Policy

**Configuration:**
```
Policy Name: "Require MFA for All Privileged Access"
Type: Privileged Access Management
Scope: Database Administrators group
Control: Multi-Factor Authentication

MFA Options:
- Method: TOTP (Time-Based One-Time Password)
- Fallback: Email OTP
- Challenge: "Enter your TOTP code from authenticator app"
- Duration: Valid for 30 seconds
```

**Expected Result:** Policy created and shown as "Active"

#### Step 4: Create Approval Workflow

**Action:** ISC Governance → Workflows → New Workflow

**Configuration:**
```
Workflow Name: "DBA Access Approval"
Trigger: Privileged access request to Database group
Step 1: Request submitted by user
Step 2: Manager approval (IT Manager routing)
Step 3: If approved → Grant access with MFA
Step 4: If denied → Send rejection notification

Manager: Morgan Chen (IT Manager)
Decision Time: 4 hours (auto-deny if not decided)
Access Duration: 2 hours if approved
```

**Expected Result:** Workflow created and enabled

#### Step 5: Test Privileged Access Request

**Action:** ISC My Requests → Request Privileged Access

**Scenario (as Casey Kim, Finance Analyst):**
```
Reason: "Need database access to restore backup"
Account: db_admin_prod (Production DBA)
Duration: 2 hours
Justification: "Production backup needs verification"
```

**Expected Result:**
```
✓ Request submitted
✓ Status: Pending Manager Approval
✓ Approver: Morgan Chen
✓ Deadline: 4 hours
```

#### Step 6: Manager Approval

**Action (as Morgan Chen):** ISC Management Console → Pending Approvals

**Approval Flow:**
```
Request from: Casey Kim
Account: db_admin_prod
Reason: Restore backup
Action Options: Approve / Deny / Request Info

Decision: APPROVE
Submit Approval
```

**Expected Result:**
```
✓ Request approved
✓ Notification sent to Casey
✓ Next: MFA challenge
```

#### Step 7: Grant Access with MFA

**Action:** ISC Session → Privilege Session Initiated

**MFA Challenge:**
```
"To access db_admin_prod, enter your TOTP code"
[User enters 6-digit code from authenticator app]
✓ Authentication successful
```

**Expected Result:**
```
Session Started: 2026-03-06 10:30:00
Account: db_admin_prod
Duration: 2 hours (expires 12:30)
Session Recording: ENABLED
Audit: All commands logged
```

#### Step 8: View Session Recording

**Action:** ISC Audits → Session Recordings → db_admin_prod / 2026-03-06

**Recording Content:**
```
10:30:15 - Login to database
10:30:45 - Execute: SELECT COUNT(*) FROM users;
10:31:12 - Execute: BACKUP DATABASE finance;
10:31:45 - Execute: RESTORE DATABASE finance FROM '/backup/';
10:32:00 - Logout

Accessibility: Playback video of all typed commands
Compliance: Evidence of who accessed what, when
```

**Expected Result:** Full session recording available with timestamps

#### Step 9: Create Emergency Access Policy

**Action:** ISC Administration → Emergency Access → Configure

**Configuration:**
```
Policy: Emergency Break-Glass Access
Trigger: User selects "Emergency Access" option
Escalation: Route to CTO immediately
MFA: Required (fastest method - SMS)
Approval: Single approver (no delay)
Duration: 1 hour maximum
Recording: ENABLED (mandatory)
Post-Access: Audit review required

Eligible Users: IT team members only
Access Level: Full database admin
```

**Expected Result:** Emergency access policy configured

#### Step 10: Test Emergency Access

**Action:** ISC Emergency Access → Request Break-Glass

**Scenario:**
```
User: Morgan Chen (IT Manager)
Reason: Production database down, normal access path broken
Account: db_admin_prod
```

**Expected Result:**
```
✓ CTO receives emergency approval request
✓ CTO can approve/deny instantly
✓ If approved: MFA required, granted within seconds
✓ Session expires in 1 hour automatically
✓ Full audit trail recorded for post-incident review
```
