## 📚 CORE CONCEPTS

### Multi-Tenancy and Isolation

**Definition:** ISC is multi-tenant SaaS. Thousands of customers on shared infrastructure, completely isolated from each other.

**Technical isolation:**
- Database isolation: Each tenant's data in separate schema/database
- API isolation: Tenant A's API calls can't access tenant B's data
- Credential isolation: Credentials for tenant A systems never visible to others
- Audit isolation: Each tenant's audit logs private

**Example:** Contoso's ISC instance and Bank's ISC instance:
- Share same ISC servers/infrastructure (cost efficiency)
- Completely separate databases (data isolation)
- Contoso can't see Bank's identities, accounts, or configuration
- Bank can't see Contoso's data
- Both encrypted and secured

---

### Authentication in ISC

**User authentication:** Login to ISC

- **Multi-factor authentication (MFA):** Required for admin accounts, optional for users
- **Single Sign-On (SSO):** Integrate with Entra ID, Okta, etc. (optional)
- **Password policy:** Minimum length, complexity, expiration (configurable)
- **Session management:** Login timeout, session expiration

**Service authentication:** APIs and connectors

- **API tokens:** Bearer tokens for REST API calls (covered in 1.17)
- **Connector credentials:** Credentials to connect to source/target systems (separate per connector)

---

### Authorization in ISC

**RBAC in ISC:** Admin roles control what ISC features people can use

- **Admin:** Full ISC access, can configure everything
- **Governance:** Can run access reviews and certifications
- **Compliance Officer:** Can access audit logs and compliance reports
- **Manager:** Can review own team's access
- **User:** Can submit access requests, view own access

Example: Contoso's Finance Manager role in ISC
- Can view Finance team's identities and accounts
- Can run access certifications
- Can NOT access HR team data or IT configuration
- Can NOT modify ISC settings

---

### Data Encryption

**Data in transit:** All communication encrypted (HTTPS/TLS 1.2+)

- API calls: Encrypted
- Connector communication: Encrypted
- Browser to ISC: Encrypted
- Virtual Appliance outbound: Encrypted HTTPS only

**Data at rest:** All data encrypted in database

- Identity data encrypted
- Credentials encrypted (never stored in plain text)
- Encryption keys managed by SailPoint (or customer-managed BYOK)

**Credentials:** Especially protected

- System credentials (for connectors) never visible in ISC UI after initial entry
- Credentials decrypted only when needed (for connection)
- Never logged or audited (prevent leaks)

---

### Audit and Compliance

**Audit trail:** All actions logged

- Who accessed ISC
- What configuration changes made
- What access provisioned/revoked
- When governance actions occurred
- Failed login attempts

**Retention:** Audit logs retained per policy (usually 1-7 years)

**Access:** Only authorized users can view audit logs

---

### Tenant Isolation Verification

**SailPoint verification:**
- Regular security audits
- Penetration testing
- Compliance certifications (SOC 2, ISO 27001, etc.)
- Customer data isolation tests

**Customer verification:**
- Audit logs show only own actions
- Search results show only own data
- API calls return only own data
- No cross-tenant data visible

---

## 🧠 KEY TAKEAWAYS

- **Multi-tenancy:** Shared infrastructure, complete data isolation
- **Authentication:** Users login with MFA, services use API tokens
- **Authorization:** RBAC controls ISC feature access (admin, governance, manager, user)
- **Encryption:** All data in transit (HTTPS) and at rest (encrypted database)
- **Isolation:** Your data completely separate from other customers

## 🎓 CERTIFICATION ALIGNMENT

**Question 1:** Contoso and a competitor both use ISC on shared infrastructure. Is there a risk of data leakage?

A) Yesshared infrastructure means data might leak
B) ✅ NoISC's multi-tenancy ensures complete isolation. Data is encrypted and isolated by tenant.
C) Yesunless each customer has own servers
D) Depends on the competitor

**Answer: B.** Multi-tenancy is specifically designed for this. Complete isolation is the whole point.

---

**Question 2:** ISC admin needs to query the database directly to recover a deleted identity. What's the risk?

A) Noneadmins can access anything
B) ✅ Risk of encryption key exposure, credential leaks, audit trail bypass. Direct DB access should avoid unless absolutely necessary.
C) Only possible with SailPoint help
D) No risk if admin has right password

**Answer: B.** Even for admins, direct database access should be avoided (bypass controls, encryption, audit). ISC APIs are safer.
