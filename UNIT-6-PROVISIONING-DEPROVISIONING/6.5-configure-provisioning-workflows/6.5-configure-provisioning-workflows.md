### TASK 2: Configure QB Provisioning Workflow

**Workflow 1: QB Finance Manager Provisioning**

```
Step 1: Create Workflow
в”ңв”Җ Name: QB_Finance_Manager_Provisioning
в”ңв”Җ Description: Provision Finance Manager to QB with admin access
в””в”Җ Status: Active

Step 2: Define Trigger
в”ңв”Җ Event: User assigned to Finance_Manager role
в”ңв”Җ Condition: Department = Finance (should be true)
в””в”Җ Action: Run this workflow

Step 3: Define Provisioning Actions
в”ңв”Җ Action 1: Create QB Account
в”Ӯ в”ңв”Җ Action: Create Account in QB
в”Ӯ в”ңв”Җ Username: [ISC: email.prefix] (casey вҶ’ casey)
в”Ӯ в”ңв”Җ Email: [ISC: email]
в”Ӯ в”ңв”Җ Password: Generate random 16 chars, send via secure email
в”Ӯ в””в”Җ Full Name: [ISC: name]
в”Ӯ
в”ңв”Җ Action 2: Set QB Permissions
в”Ӯ в”ңв”Җ Action: Add to QB permission group
в”Ӯ в”ңв”Җ Permission level: "admin"
в”Ӯ в”ңв”Җ QB groups: ["Accounting", "Management", "Reports"]
в”Ӯ в””в”Җ GL access: enabled
в”Ӯ
в”ңв”Җ Action 3: Send Notification
в”Ӯ в”ңв”Җ Recipient: User email
в”Ӯ в”ңв”Җ Subject: "Your QuickBooks Admin Access is Ready"
в”Ӯ в”ңв”Җ Body: "Casey, your QB account is active. Temp password: [password]. Please change on first login."
в”Ӯ в””в”Җ Send: Secure email
в”Ӯ
в”ңв”Җ Action 4: Log Provisioning
в”Ӯ в”ңв”Җ Log: "User [Casey] provisioned to QB [admin], Date: [today], Result: Success"
в”Ӯ в””в”Җ Audit trail: Complete

Step 4: Error Handling
в”ңв”Җ If QB account already exists:
в”Ӯ в””в”Җ Action: Use existing account, set permissions (don't create duplicate)
в”ңв”Җ If QB API timeout:
в”Ӯ в”ңв”Җ Action: Retry 3 times
в”Ӯ в”ңв”Җ Wait: 30 seconds between retries
в”Ӯ в””в”Җ Alert: Notify admin if all retries fail
в”ңв”Җ If user missing required attributes (email):
в”Ӯ в””в”Җ Action: Fail and alert admin
в””в”Җ Result: Documented, admin can manually fix

Step 5: Verification
в”ңв”Җ After provisioning, verify:
в”Ӯ в”ңв”Җ Account created in QB вң“
в”Ӯ в”ңв”Җ Permission level set to admin вң“
в”Ӯ в”ңв”Җ User can log in вң“
в”Ӯ в””в”Җ Audit logged вң“
```

## рҹҺ“ CERTIFICATION

**Q:** What should happen if QB provisioning fails due to API timeout?

A) Skip and continue
B) вң… Retry 3 times, then alert admin for manual intervention
C) Disable user's role
D) Delete user and start over

**Answer: B.** Failed provisioning requires resolution - user shouldn't lose access, admin must fix manually if retries fail.

**Q:** When configuring GitHub provisioning for Engineer_Developer, what permission level should they have?

A) Admin (can manage repos)
B) вң… Contributor (can push code, create PRs, but not delete repos)
C) Read-only (can view code only)
D) No access (manual approval needed)

**Answer: B.** Engineer_Developer = contributor level (write access to dev/staging, no admin).
